Challenges and Lessons Learned with Jupyter Notebooks and GitHub Pages

Reflecting on the hurdles faced while building and sharing projects using Jupyter Notebooks and GitHub Pages.

1. Rendering Jupyter Notebooks on GitHub Pages

What I learned:

  • I had to rely on tools like nbconvert to convert notebooks into HTML, which required tweaking the templates and ensuring proper styling to match the rest of the website.
  • Managing multiple notebooks with consistent styles and interactivity was a significant effort, requiring me to standardize the templates and ensure the presentation remained uniform.

What I would do differently:

  • I would build a more robust solution for automating the styling of converted notebooks to ensure uniformity in the presentation without needing to manually edit templates for each project.
  • I’d consider using a more interactive notebook hosting service like Binder or Voila, which would allow me to provide a richer, more dynamic experience for users instead of static HTML.

2. Integration with GitHub Pages and Jupyter Notebooks

What I learned:

  • I had to structure my repository efficiently, ensuring notebooks, static assets (like images), and HTML files were all in the correct directories for GitHub Pages to serve them properly.
  • I also had to manually handle updates, which was tedious, and required frequently checking the status of notebooks and converting them into HTML format when making changes.

What I would do differently:

  • I would automate the process of pushing changes to GitHub Pages using GitHub Actions, which would automatically convert the notebook to HTML and push the changes to the live site without manual intervention.
  • I’d also make better use of version control to track changes in the notebooks and ensure that the GitHub Pages site always displays the latest version.

3. Lack of Development Environment

What I learned:

  • Since GitHub Pages does not provide a development environment for Jupyter Notebooks, I had to commit each change to trigger the site updates. This created inefficiencies and added extra steps, as every minor modification in the notebooks required a commit to update the live site.

What I would do differently:

  • I would explore more flexible hosting solutions that allow live development and provide automatic rendering and previewing of notebooks, like Binder or Voila, to reduce the friction of manual commits for every change.
